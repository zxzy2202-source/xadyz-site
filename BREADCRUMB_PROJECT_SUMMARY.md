```
│ │ main                                 │ │
│ │ ┌─────────────────────────────────┐ │ │
│ │ │ Hero Section (自定义)           │ │ │
│ │ └─────────────────────────────────┘ │ │
│ │ ┌─────────────────────────────────┐ │ │
│ │ │ Breadcrumb (自动插入) ← 核心！ │ │ │
│ │ └─────────────────────────────────┘ │ │
│ │ ┌─────────────────────────────────┐ │ │
│ │ │ Page Body (正文内容)            │ │ │
│ │ └─────────────────────────────────┘ │ │
│ └─────────────────────────────────────┘ │
│ ┌─────────────────────────────────────┐ │
│ │ Footer (页脚)                        │ │
│ └─────────────────────────────────────┘ │
└─────────────────────────────────────────┘
```

---

### **数据流：**

```
URL (/en/products/thermal-paper-rolls)
  ↓
window.location.pathname
  ↓
Breadcrumb 组件
  ↓
解析 URL segments: ["products", "thermal-paper-rolls"]
  ↓
匹配 routeTree: products → thermal-paper-rolls
  ↓
获取翻译 crumbI18n[lang]: "Home › Products › Thermal Paper Rolls"
  ↓
渲染面包屑 UI
```

---

### **单一真相源：**

```
src/seo/routeTree.ts (路由结构)
         ↓
    ┌────┴────┐
    ↓         ↓
Breadcrumb  SEO JSON-LD
 组件       结构化数据
```

**优势：**
- ✅ 修改一次，全局同步
- ✅ 减少维护成本
- ✅ 保证一致性

## 🔧 技术决策

### **决策 1: 为什么选择 PageShell？**

**背景：** 项目没有统一 Layout，每个页面独立写结构。

**选项对比：**

| 方案 | 优点 | 缺点 | 结果 |
|------|------|------|------|
| A. 每个页面手动插入 | 快速、简单 | 代码重复、维护困难 | ❌ 放弃 |
| B. 创建 PageShell | 代码优雅、易维护 | 改动较大 | ✅ **采纳** |
| C. HOC 高阶组件 | 灵活 | 复杂度高、不直观 | ❌ 放弃 |

**最终决策：** 方案 B（PageShell）

**原因：**
1. 长期维护成本最低
2. 代码可读性最好
3. 改造成本可接受（单页 5-10 分钟）
4. 为未来功能扩展打下基础

---

### **决策 2: 如何处理 Hero 覆盖问题？**

**问题：** Hero Section 使用 `absolute inset-0` + `-mt-32`，会覆盖下方内容。

**解决方案：**

```tsx
// PageShell.tsx
<div className="relative z-10 max-w-7xl mx-auto px-4 -mt-10 mb-8">
  <Breadcrumb lang={lang} />
</div>
```

**技术细节：**
- `relative z-10`：提高层级，浮在 Hero 上方
- `-mt-10`：负边距，让面包屑贴近 Hero 底部
- 如果仍被覆盖，可增加 z-index 或添加白色背景

---

### **决策 3: lang 参数传递方式**

**选项 1：** 从 pathname 自动推断
```tsx
// ❌ 不稳定，容易出错
const lang = pathname.split('/')[1];
```

**选项 2：** 显式传递 lang 参数
```tsx
// ✅ 明确、可控、类型安全
<Breadcrumb lang={lang} />
```

**最终决策：** 选项 2

**原因：**
1. 类型安全（TypeScript 强制检查）
2. 不依赖 URL 格式
3. 易于测试和调试

## 🚨 风险和注意事项

### **风险 1: Hero 覆盖问题**

**可能性：** 中  
**影响：** 高（面包屑不可见）

**缓解措施：**
- ✅ 使用 `z-index` 提高层级
- ✅ 添加负边距调整位置
- ✅ 必要时添加白色背景
- ✅ 提供多个调试方案

## 🎓 经验总结

### **成功因素：**

1. ✅ **单一真相源设计**
   - routeTree.ts 集中管理所有路由
   - 修改一次，全局同步
   - 大幅降低维护成本

2. ✅ **组件化和复用**
   - PageShell 提供统一结构
   - Breadcrumb 自动化生成
   - 减少重复代码

3. ✅ **完整的文档系统**
   - 改造清单、模板、测试用例
   - 降低开发人员上手难度
   - 提高交付效率

4. ✅ **渐进式集成策略**
   - 先完成核心组件
   - 示例页面验证
   - 再批量推广
   - 降低风险

---

### **教训：**

1. ⚠️ **应该更早发现 Layout 缺失**
   - 项目初期应该先设计统一 Layout
   - 避免后期大规模重构

2. ⚠️ **Hero 覆盖问题应提前预防**
   - 绝对定位 + 负边距容易导致问题
   - 设计时应考虑后续内容插入

3. ⚠️ **多语言参数传递应统一**
   - 从 pathname 推断不可靠
   - 显式传递更稳定
